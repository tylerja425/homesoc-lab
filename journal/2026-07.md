### SSH key authentication — set up, with two real mistakes along the way
Generated an ED25519 keypair on the Windows machine:
```
ssh-keygen -t ed25519 -C "security-onion-lab"
```
Saved to the default location, set with a passphrase (decided a bare,
no-passphrase key wasn't worth the reduced security for convenience —
went with KeePassXC's SSH Agent feature instead, so the passphrase is
stored and auto-supplied without retyping it each time).

**KeePassXC SSH Agent setup:**
- Tools → Settings → SSH Agent → enabled, set to "Use OpenSSH" (not
  Pageant — Pageant is PuTTY's agent protocol; PowerShell's native `ssh`
  expects the Windows OpenSSH agent instead).
- Added the private key (`id_ed25519`, **not** the `.pub` file — the
  private key is what the agent needs to actually authenticate) to an
  entry's SSH Agent tab, with "Add key to agent when database is
  opened/unlocked" and "Remove key from agent when database is
  closed/locked" both checked.

**Mistake 1 — disabled the wrong service.** Earlier in this session,
disabled Windows' built-in `ssh-agent` service (OpenSSH Authentication
Agent) on the assumption it would conflict with KeePassXC. Backwards:
KeePassXC's "Use OpenSSH" mode doesn't replace that service, it feeds
keys *into* it. Without the service running, there's no agent pipe for
either KeePassXC or `ssh` to use at all. Fixed by re-enabling it from an
elevated PowerShell:
```
Set-Service -Name ssh-agent -StartupType Automatic
Start-Service ssh-agent
```
After starting the service, had to lock and re-unlock the KeePassXC
database to get it to push the key into the now-running agent — confirmed
with `ssh-add -l`, which showed the key's fingerprint once it worked.

**Mistake 2 — never actually copied the public key to the server.** Even
with the agent working locally (`ssh-add -l` showing the key), SSH kept
falling back to password auth. Turned out the earlier command to copy the
public key into the VM's `authorized_keys` had been given but never
confirmed run — got sidetracked into the KeePassXC setup before checking.
Verified directly on the server:
```
cat ~/.ssh/authorized_keys
# → No such file or directory
```
Ran the actual copy command (from the Windows side, not the SSH session):
```
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh sysop@192.168.8.12 "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```
One last password prompt to do the copy, then `ssh sysop@192.168.8.12`
logged straight in with no password. Confirmed working end to end.

**Lesson:** when chaining several setup steps together, verify each one
actually completed before moving to the next — two separate assumptions
(that a command had been run, and that a service should be disabled)
both turned out wrong and combined to make the actual root cause harder
to spot.

### Kali install finished — static IP didn't persist
Install completed, GRUB installed to primary drive (only OS on the disk,
so straightforward). First login confirmed. `ip a` showed `eth0` up but
with **no IPv4 address** — the static `10.10.10.10/24` config entered
during the installer didn't carry over to the installed system.

Checked `/etc/network/interfaces` — only had loopback configured, nothing
for `eth0`. Turned out this system manages networking via NetworkManager
(`systemctl status NetworkManager` → active), not the classic
`interfaces` file, so the installer's static IP settings likely only
applied to the live installer session and never got written into
NetworkManager's config for the installed system.

Fixed directly with `nmcli` instead of fighting the installer:
```
sudo nmcli connection modify "Wired connection 1" ipv4.addresses 10.10.10.10/24 ipv4.method manual
sudo nmcli connection up "Wired connection 1"
```
Confirmed with `ip a` — `eth0` now shows `10.10.10.10/24` correctly, and
it persisted through a later reboot.

## 2026-07-18 — Metasploitable2, and a real multi-layered detection bug hunt

### SSH to the Proxmox host itself
Before starting on Metasploitable, set up key auth to SV1 (the Proxmox
host) too, not just the VMs — reused the same ED25519 keypair. `root@SV1`
accepted password auth already; copied the public key over the same way
as before and confirmed key login worked. Flagged as a genuinely
sensitive credential (root on the hypervisor underlies every other VM),
so extra emphasis on keeping the KeePassXC database itself well protected
going forward.

### Metasploitable2 download and import
Official source confirmed: SourceForge, `metasploitable-linux-2.0.0.zip`
(~800MB), default creds `msfadmin`/`msfadmin`. Per the project's own
warning, never expose this VM to an untrusted network — a non-issue here
since it lives on the isolated `vmbr1` segment by design, same as Kali.

Downloaded and extracted directly on the Proxmox host via SSH:
```
cd /var/lib/vz/template/iso
wget https://sourceforge.net/projects/metasploitable/files/Metasploitable2/metasploitable-linux-2.0.0.zip/download -O metasploitable-linux-2.0.0.zip
apt install -y unzip   # not installed by default on Proxmox
unzip metasploitable-linux-2.0.0.zip
```
Extracted into `Metasploitable2-Linux/`, containing `Metasploitable.vmdk`
(the actual disk image) plus some VMware-specific files that aren't
needed for Proxmox.

Created VM 104 (`metasploitable`) with no disk (deleted the auto-created
one), 1 core, 512MB RAM, network on `vmbr1`, QEMU agent checked at
creation. Imported the VMDK as a new disk:
```
qm importdisk 104 Metasploitable.vmdk local-lvm
```
Attached the resulting disk via the Hardware tab (auto-assigned as
`scsi0`), boot order auto-updated to boot from it.

### Bug 1 — wrong disk controller for a 2012-era kernel
First boot failed: `ALERT! /dev/mapper/metasploitable-root does not
exist. Dropping to a shell!`, and `cat /proc/partitions` inside the
rescue shell showed **no disks detected at all**. Root cause: Proxmox's
default SCSI controller for this VM was VirtIO SCSI, which didn't exist
as a common option when this VM image was built (~2012) — its old
kernel/initramfs has no driver for it, so it couldn't see the disk to
even attempt mounting it.

Fixed by moving the disk from `scsi0` to an IDE bus instead (universally
supported, even by very old kernels), via the Proxmox shell rather than
the GUI (no in-place bus-conversion option in the UI, only detach/
reattach):
```
qm set 104 --ide0 local-lvm:vm-104-disk-0
qm set 104 --delete scsi0
qm set 104 --boot order=ide0
```
Booted cleanly after this — full ASCII banner, services starting
(Tomcat, Apache), login prompt reached.

### Networking: same static-IP pattern, older tooling
Logged in (`msfadmin`/`msfadmin`), `eth0` had no IPv4 (only link-local
IPv6) despite being up — this OS predates NetworkManager/netplan
entirely, so it just needed the classic Debian
`/etc/network/interfaces` file edited directly:
```
auto eth0
iface eth0 inet static
    address 10.10.10.20
    netmask 255.255.255.0
```
`sudo ifdown eth0 && sudo ifup eth0` applied it. Confirmed with
`ifconfig eth0` (this old system doesn't have the modern `ip` command)
— `inet addr:10.10.10.20` correct, and RX/TX counters showed it was
actively passing traffic.

### Connectivity confirmed — but nothing showed up in Security Onion
`ping -c 4 10.10.10.20` from Kali: 0% packet loss, working perfectly.
But searching Security Onion's Hunt page for `10.10.10.20` after running
an nmap scan (`nmap -sV 10.10.10.20`) returned **zero results**. This
kicked off a real, multi-layered troubleshooting chain — worth writing
up in full since each layer looked like "the" answer before turning out
to be one piece of a bigger picture.

**Layer 1 — search technique, ruled out.** A bare IP as a Hunt query is
valid Onion Query Language (matches across all fields), confirmed this
was working syntax; it just wasn't matching anything real. A plain
`10.10.10.20` search briefly returned results that turned out to be
Security Onion's *own* web dashboard API logs (`event.dataset:
soc.server`) — the browser's own traffic to the SOC interface, not
network capture data. Switched to a more specific query
(`source.ip:10.10.10.10 OR destination.ip:10.10.10.20`) to avoid that
noise going forward.

**Layer 2 — the sniffing interface was never actually active.**
Checked `ip a` on Security Onion directly: `ens19` (the monitor NIC) was
`UP` at the link level but missing the `PROMISC` flag, and `bond0` (the
bonded interface Security Onion actually captures from — this is normal
Security Onion design, not a misconfiguration) showed `NO-CARRIER` /
`state DOWN`. `nmcli con show` revealed why: a connection profile named
`bond0-slave-ens19` existed (the intended binding) but was never
activated — `DEVICE` column showed `--`.

Tried activating it manually (`sudo nmcli con up bond0-slave-ens19`) —
failed with `Error: Connection activation failed: Unknown error`. This
is the **exact same error text** seen during the original install (and
dismissed at the time as a false alarm after `so-status` came back
clean) — in hindsight, that dismissal may have been premature; this may
be the same underlying issue surfacing again rather than a one-off.

Checked `journalctl -u NetworkManager` for the real reason this time
(the connection-specific filter returned nothing useful, but a broader
`--since "5 minutes ago"` filter did): `mtu greater than device maximum
... setting MTU to requested size is not possible`. Security Onion's own
config (`/opt/so/saltstack/local/pillar/minions/securityonion_
standalone.sls`) specifies `mtu: 9000` for the monitor interface (jumbo
frames — matches Security Onion's own documented Proxmox guidance), but
`vmbr1` and Security Onion's `net1` NIC were still at Proxmox's default
1500, so the bond-slave attachment kept failing at the MTU-setting step.

Fixed by setting `vmbr1`'s MTU to 9000 in Proxmox (Network → vmbr1 →
Edit), and leaving Security Onion's `net1` NIC set to "Same as bridge"
(inherits 9000 automatically, avoids the two settings drifting out of
sync later). **This required a full VM restart** to take effect — a
live Proxmox-side MTU change doesn't reach an already-running guest, same
lesson as the QEMU guest agent issue from the initial install. After
restart: `ens19` showed `mtu 9000`, and `nmcli con up bond0-slave-ens19`
succeeded — `ens19` now shows `PROMISC` and `master bond0`, `bond0` shows
`UP, LOWER_UP`.

**Layer 3 — sniffing was active, but still saw almost nothing.** With
the bond genuinely up, re-ran the ping/nmap test. `tcpdump -i bond0 -n`
on Security Onion captured exactly one packet (an ARP broadcast) and
some unrelated DHCP broadcast noise from Metasploitable — but *zero* of
the actual ICMP ping traffic between Kali and Metasploitable, even
though the ping itself succeeded perfectly (0% loss) on Kali's end.

This is the real structural gap, and it's about how Linux bridges work,
not a Security Onion problem: `vmbr1` is a standard Linux bridge, which
forwards unicast traffic directly port-to-port based on MAC learning —
it does **not** flood unicast traffic to every port the way a hub would.
Being in promiscuous mode only helps Security Onion's port see whatever
traffic *actually arrives* there; a two-way conversation between Kali's
and Metasploitable's ports simply never gets forwarded to a third,
uninvolved port unless something explicitly mirrors it there. This is
the fundamental reason a plain bridge isn't sufficient for a "sniffing"
setup — real SPAN/port-mirror configuration is a deliberate, additional
step, not a side effect of promiscuous mode.

Fixed with explicit Linux `tc` (traffic control) mirroring rules on the
Proxmox host, mirroring both VMs' tap interfaces to Security Onion's tap
interface. Identified the actual host-side tap device names first
(Proxmox's `tap<VMID>i<index>` convention):
```
ip a | grep -B 1 "master vmbr1"
# → tap103i0 (Kali), tap104i0 (Metasploitable), tap101i1 (Security Onion)
```
Then added mirroring rules on both VM ports, targeting Security Onion's
port:
```
tc qdisc add dev tap103i0 handle ffff: ingress
tc filter add dev tap103i0 parent ffff: protocol all u32 match u32 0 0 action mirred egress mirror dev tap101i1

tc qdisc add dev tap104i0 handle ffff: ingress
tc filter add dev tap104i0 parent ffff: protocol all u32 match u32 0 0 action mirred egress mirror dev tap101i1
```
Retested immediately: `tcpdump -i bond0 -n host 10.10.10.20` on Security
Onion now showed the full ICMP conversation (all 4 echo request/reply
pairs) while pinging from Kali. Confirmed working.

**Note for later:** these `tc` rules are applied directly via the Proxmox
shell and are **not persistent** — they won't survive a host reboot.
Should be scripted/automated (e.g. a boot-time hook or Proxmox hookscript)
if this lab is going to be rebuilt or the host restarted; for now,
documenting the exact commands here is the fallback if they need to be
reapplied manually.

### End-to-end confirmed
Re-ran `nmap -sV 10.10.10.20` from Kali and searched Security Onion's
Hunt page (`source.ip:10.10.10.10 OR destination.ip:10.10.10.20`).
Real results this time: dozens of `zeek.conn` and `zeek.weird` entries
across every port nmap probed, and a genuine Suricata alert —
**"ET SCAN Suspicious inbound to PostgreSQL port 5432"** (category:
Potentially Bad Traffic) — triggered by the scan hitting Metasploitable's
PostgreSQL port. The full detection pipeline (isolated network → traffic
generation → mirroring → Zeek/Suricata analysis → SOC dashboard) is
confirmed working end to end.

### Status
Home SOC lab is functionally complete: Security Onion, Kali attacker VM,
and Metasploitable2 target VM all running on the isolated `vmbr1` segment,
traffic properly mirrored to Security Onion's sniffing interface, real
alerts confirmed generating and visible in the SOC dashboard.

## 2026-07-18 (cont'd) — Making the mirroring rules persistent

### The problem
The `tc` mirroring rules from earlier today are attached to the VMs' tap
interfaces, but tap interfaces themselves get destroyed and recreated
every time a VM stops/starts (Proxmox tears them down on shutdown). That
means the rules silently vanish on any VM restart or host reboot, with
no error, no warning — the lab would just quietly stop capturing traffic
again until someone remembered to reapply them by hand.

### Fix: Proxmox hookscript
Proxmox supports "hookscripts" — custom scripts that run automatically
at VM lifecycle events (pre-start, post-start, pre-stop, post-stop).
Used `post-start` to reapply the mirroring rules right after a VM (and
its tap interface) actually comes up.

Enabled the `local` storage's "Snippets" content type (Datacenter →
Storage → local → Edit → Content), required before Proxmox will let a VM
reference a hookscript stored there. Created
`/var/lib/vz/snippets/mirror-hook.sh`:

```bash
#!/bin/bash
# Reapplies tc mirroring rules for the homesoc-lab isolated network
# (vmbr1) so Security Onion (tap101i1) sees traffic between Kali
# (tap103i0) and Metasploitable (tap104i0). Runs on post-start since tap
# interfaces are destroyed/recreated on every VM stop/start.

VMID="$1"
PHASE="$2"

if [ "$PHASE" != "post-start" ]; then
    exit 0
fi

if [ "$VMID" != "103" ] && [ "$VMID" != "104" ]; then
    exit 0
fi

TARGET_TAP="tap101i1"
SOURCE_TAP="tap${VMID}i0"

sleep 2  # let the tap interface fully settle

tc qdisc add dev "$SOURCE_TAP" handle ffff: ingress 2>/dev/null
tc filter add dev "$SOURCE_TAP" parent ffff: protocol all u32 match u32 0 0 action mirred egress mirror dev "$TARGET_TAP" 2>/dev/null

exit 0
```

Made it executable (`chmod +x`) and attached it to both source VMs:
```
qm set 103 --hookscript local:snippets/mirror-hook.sh
qm set 104 --hookscript local:snippets/mirror-hook.sh
```

### Verified with a real stop/start cycle
```
qm stop 103 && qm stop 104
qm start 103 && qm start 104
tc filter show dev tap103i0 parent ffff:
tc filter show dev tap104i0 parent ffff:
```
Both filters reappeared automatically, no manual `tc` commands run.
Confirmed with a fresh ping test + `tcpdump -i bond0` on Security
Onion — full ARP resolution and ICMP exchange captured immediately
after the restart, no manual intervention needed.

### Verified with a full host reboot
Rebooted the entire Proxmox host (`reboot`, not just the lab VMs — this
briefly took down the other production VMs on the same host, `srv-infra`
and `ai-workloads`, timed for a moment that didn't matter). After SV1
came back up, checked VM states with `qm list` — none of the lab VMs
auto-start by default, so all needed manually starting except
Metasploitable, which showed as still `running` (its tap interface
apparently survived the host reboot in whatever state it was in, rather
than being torn down and recreated like a normal VM restart would do).

Started Security Onion and Kali fresh (`qm start 101`, `qm start 103`).
Checked mirroring rules afterward:
- `tap103i0` (Kali, which went through a genuine stop→start cycle):
  filter present, hookscript fired correctly.
- `tap104i0` (Metasploitable, which never actually restarted): filter
  **missing** — confirms the hookscript only fires on an actual
  `post-start` event, and a VM that survives a host reboot without
  itself stopping/starting won't get the rule reapplied.

Fixed by explicitly cycling Metasploitable too (`qm stop 104` /
`qm start 104`) — filter reappeared immediately afterward. Both VMs now
confirmed with active mirroring rules following a genuine full host
reboot, not just VM-level restarts.

**Lesson:** the hookscript is reliable for "the VM actually restarts,"
but a VM that stays running through a host reboot (however that
happens) won't trigger it. Worth remembering if this lab is left running
long-term and the host ever reboots unexpectedly — check `tc filter show`
on both tap interfaces after any host-level event, don't just assume
persistence covers every scenario.

### Status
Home SOC lab is now fully and genuinely persistent, confirmed via an
actual full host reboot: Security Onion, Kali, and Metasploitable all
end up with correct mirroring in place after restarting, whether
triggered by a VM-level or host-level event (with the one caveat noted
above for VMs that don't themselves restart).

## 2026-07-18 — First real alert triage: the PostgreSQL scan alert

### Goal for this session
Actually work an alert in the SOC dashboard as an analyst would, instead
of just confirming the pipeline could generate one. Also a deliberate
teaching exercise — walking through triage reasoning step by step, not
just reaching a verdict.

### Starting point: the Alerts page
Opened the **Alerts** page (not Hunt) first, since that's where an
analyst starts triage — Alerts surfaces things Security Onion already
flagged, Hunt is for open-ended searching once you have a lead. The last
24 hours showed 14 distinct alert types, all tracing back to one real
event: the `nmap -sV 10.10.10.20` scan run from Kali during the earlier
pipeline debugging. Notable entries: two "ET SCAN Nmap ... User-Agent"
alerts (Suricata correctly fingerprinting nmap itself, marked **high**
severity, count 8 each), a "GPL ICMP PING *NIX" (the scan's ping sweep),
and a cluster of "ET SCAN Suspicious inbound to [database] port" alerts
(PostgreSQL, MySQL, MSSQL, Oracle SQL — all **medium** severity).

### Picking what to open first
Talked through how an analyst actually prioritizes a list like this,
rather than working top to bottom:
- Recognize when many alerts are fragments of one story, not separate
  incidents — this whole list was one scan producing many signature
  hits, not 14 independent problems.
- Severity labels reflect the rule author's confidence in the traffic
  pattern, not business impact — the database-port alerts (medium)
  arguably mattered more to investigate than the nmap-fingerprint
  alerts (high), since they point at what was actually being probed.
- High count often just means "matched repeatedly," not "more severe."
  A count of 1 can be more interesting than a count of 8.
- The real question worth chasing: what does an alert tell you was
  *attempted* vs. what was *accomplished*? Picked the PostgreSQL alert
  because it sat on a real, exposed service, in the middle of an obvious
  scan, medium severity, and was likely to have follow-up detail worth
  pulling.

### Reading the alert itself
Opened the PostgreSQL alert's full record. Key fields, in the order they
actually appear in the UI (noted this explicitly after finding the
alphabetical/JSON field order hard to follow against a wall of text):
`destination.ip`/`destination.port` near the top (10.10.10.20, 5432),
`source.ip`/`source.port` further down (10.10.10.10, 55680), and
`rule.rule` containing the actual Suricata rule logic.

The rule itself only checks for `flags:S` (a SYN packet) to port 5432,
thresholded at 5+ hits from the same source within 60 seconds — it's a
scan-pattern detector, not a payload/exploit detector. Also noticed
`rule.metadata.signature_severity: ["Informational"]` — even the rule's
own author rated it low severity at the signature level, and it's an old
rule (created 2010, last updated 2019).

**Lesson:** read what a rule's logic actually checks for, not just its
name. "Suspicious inbound to PostgreSQL" sounds like a database attack;
the rule itself is really just "enough SYN packets hit this port to look
like a scan."

### Corroborating with Zeek via Hunt
Searched Hunt for `source.ip:10.10.10.10 AND destination.ip:10.10.10.20
AND destination.port:5432` to see what Zeek recorded for the same
traffic — the Suricata alert alone can't say whether the connection
actually completed or did anything beyond triggering a signature match.

Found matching `zeek.conn` and `zeek.weird` entries at the exact same
timestamp and source port as the alert (confirms all three log entries
describe the same single event, not three coincidentally similar ones —
this cross-source correlation by timestamp + connection tuple is the
actual technique, not just "looking at nearby events").

The `zeek.conn` entry showed `connection.state: SHR` with a plain
-English description: "Responder sent a SYN ACK followed by a FIN, we
never saw a SYN from the originator" — `client.bytes`/`client.packets`
both 0. Initially guessed this might be a mirroring artifact (our tap
mirroring setup isn't a perfectly synchronized single capture point), but
the paired `zeek.weird` entry explained it precisely instead:
`weird.name: active_connection_reuse` — a documented Zeek anomaly type
meaning a new SYN reused the same source/destination IP+port tuple as a
connection Zeek hadn't finished aging out yet. This is a well-known,
benign side effect of fast port scanning, not a mirroring flaw — nmap's
rapid-fire probes reusing ports faster than Zeek's connection tracking
can cleanly close out prior entries.

### Verdict
True positive for reconnaissance/scanning (nmap against Metasploitable's
PostgreSQL port), confirmed across two independent log sources
(Suricata + Zeek). No evidence of a completed connection, successful
login, or data access — the "weird" connection-state artifact is
explained by scan speed, not by anything unusual happening beyond the
scan itself. In a real SOC this would close as a confirmed, authorized
scan (since it's known lab activity) or, if unauthorized, escalate as
reconnaissance warranting increased monitoring on the target host.

### On checking the other alerts (MySQL, MSSQL, Oracle, RPC, etc.)
Talked through the actual decision rule for whether it's worth
fully triaging similar-looking alerts once one is understood:
- Spot-check 1-2 to confirm the pattern holds; don't re-derive the same
  conclusion 6 times once confidence is high.
- Worth a full look anyway if: the Zeek outcome differs (a completed
  connection vs. another half-open scan probe), the severity/rule
  category differs in kind (not just "another SCAN hit"), or the
  timestamp sits outside the tight cluster of the rest of the scan.
- Didn't do a full pass on the remaining alerts this session — noted as
  a natural next step if/when picking this back up.

### Reflection
This was also a deliberate exercise in explaining triage reasoning step
by step rather than just stating conclusions — including correcting
course mid-exercise (the mirroring-artifact guess turned out wrong once
the actual `weird.name` field was checked) and adjusting how information
was presented after the field-order approach was confusing to follow.
Good direct mapping to CySA+ domain material: this whole exercise sits
squarely in Security Operations (tool-based investigation, correlating
SIEM/NSM data sources) and touches Incident Response (triage, evidence
gathering, reaching a verdict).

### Next steps

  that doesn't.

## 2026-07-18 — Disabling SSH password authentication (all hosts)

### Goal for this session
Close out a long-standing to-do: disable SSH password auth on every host
where key-based auth was already confirmed working, so key auth becomes
the only way in. Covers Proxmox (the hypervisor itself), Security Onion,
and Kali. Metasploitable was intentionally skipped — it has no
home-network interface at all, so this doesn't apply to it.

### Confirmed key auth first
Before touching any config, confirmed key-based SSH already worked on
all three hosts (via KeePassXC's SSH agent) — important, since disabling
password auth on a host without working key auth would risk a lockout.

### The edit
Same change needed on all three: find `PasswordAuthentication` in
`/etc/ssh/sshd_config`, uncomment it, and set it to `no`. All three hosts
had the identical default line: `#PasswordAuthentication yes`.

### The vi detour (worth remembering)
Started editing Security Onion's config with `vi` (no `nano` available —
Security Onion is Rocky Linux under the hood, minimal install). Ran into
real trouble here, worth documenting since it's a genuinely common
beginner trip-up: vi has two modes, **normal mode** (letters are
commands — `x` deletes a character, `dd` deletes a line) and **insert
mode** (letters are typed as literal text), and it's easy to lose track
of which one you're in. Arrow key movement while still in insert mode
led to text getting typed in the wrong place, and a `dd`-then-retype
attempt ended up deleting the line's value entirely rather than editing
it cleanly.

**Recovery:** checked the file with `grep -n PasswordAuthentication
/etc/ssh/sshd_config` and found the line was actually still in its
original, untouched state — the botched vi edits had never actually been
saved. No damage done, just wasted motion.

**Switched approach entirely** — used `sed` instead of an interactive
editor for this specific one-line change, since it's a single, exact,
scriptable edit with no modes or cursor position to track:

```
sed -i 's/^#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
```

This worked cleanly on the first try on all three hosts. **Lesson:** for
a single well-defined line change, `sed` is safer and faster than
opening an interactive editor, especially one you're not fluent in yet.
Interactive editors are still worth learning, but the right tool for a
one-line, unambiguous edit is often not an editor at all.

### Applying and verifying, host by host
For each host: made the `sed` edit, restarted the SSH service **without
closing the already-open terminal session** (restarting the service
doesn't drop existing connections, only affects new ones), then opened a
brand new terminal and tested a fresh key-based connection before
considering that host done. This way, if a restart ever broke something,
the original session would still be there as a fallback.

- **Proxmox (root@192.168.8.10):** service name `ssh` (Debian-based).
  Edited, restarted, fresh connection succeeded with no password prompt.
- **Security Onion (sysop@192.168.8.12):** service name `sshd` (Rocky
  Linux naming differs from Debian). Edited, restarted, fresh connection
  succeeded.
- **Kali (tyler@192.168.8.13):** service name `ssh` (Debian-based).
  Edited, restarted, fresh connection succeeded.

### Status
Password authentication is now disabled on all three home-network-facing
hosts in the lab (Proxmox, Security Onion, Kali) — key-based auth is the
only way in on any of them, confirmed working post-restart on each.
Metasploitable is unaffected (no home-network exposure by design).

This closes out the last item that had been sitting on the lab's to-do
list since the original Security Onion install.

### Goal for this session
Add a vulnerability scanner (Greenbone Community Edition / OpenVAS) to the
lab, run a real scan against Metasploitable, and — once a finding turned
up — actually verify it by exploiting it, then check whether Security
Onion caught the exploitation happening.

### New VM: vuln-scanner
Built VM 105 the same way as the others: Debian 13 (trixie — Debian 12
"bookworm" is no longer current; confirmed via Debian's own SHA256SUMS
before installing), 4 cores, 8GB RAM, 60GB disk, dual-homed on `vmbr0`
(`192.168.8.14`, DHCP reservation) and `vmbr1` (`10.10.10.30`, static).
SSH key auth set up the same way as every other host (reused the existing
key already trusted elsewhere rather than generating a new one), and
password auth disabled to match the rest of the lab.

Installed Docker Engine + Compose plugin from Docker's official repo
(confirmed it actually supports Debian 13 — `trixie` is listed at
`download.docker.com/linux/debian/dists/` — before assuming it would
work). Deployed Greenbone Community Edition via its official Docker
Compose file. One hiccup: the `scap-data` container initially came up
"unhealthy" even though its logs showed the CVE/CPE copy job had
genuinely finished — a health-check timing issue, not a real failure;
retrying `docker compose up -d` fixed it on the second pass (same
"looked like a failure, wasn't" pattern as the original Security Onion
install).

Greenbone's web UI binds to `127.0.0.1` only by default (safe-by-default
design). Decided to expose it directly on `vmbr0` instead of using an SSH
tunnel, edited `compose.yaml` to bind nginx to `0.0.0.0`, now reachable
at `https://192.168.8.14` same as every other dashboard in the lab.

First feed sync (NVTs, SCAP/CVE data, CERT advisories, GVMD data) took
about half a day in the background — expected, per Greenbone's own docs,
and confirmed genuinely in-progress (not stuck) by checking container
logs directly rather than trusting the static "Update in progress" label
alone.

### First scan against Metasploitable
Created a target (`10.10.10.20`, default port list, no authenticated-scan
credentials — first pass was intentionally unauthenticated) and ran a
"Full and fast" scan. Result: 66 findings, 118 CVEs, several Critical
(10.0) severities. Metasploitable is, as designed, extremely exploitable:
the rexec service, rlogin passwordless login, MySQL default credentials,
DistCC RCE, the Ingreslock backdoor, TWiki RCE, and — the one we chose to
dig into — the vsftpd 2.3.4 backdoor.

### Understanding the vsftpd finding
Read through the finding detail: this isn't a code bug, it's a supply
chain compromise. In July 2011, attackers replaced vsftpd's official
source archive on its own hosting server for about three days; anyone
who compiled vsftpd during that window got a version with a hidden
backdoor. Sending `:)` as an FTP username triggers it, opening a root
shell on port 6200/tcp. Good example of a vulnerability category that
isn't "someone wrote buggy code" — worth remembering for CySA+
Vulnerability Management material, which distinguishes control/risk
types like this.

### Trying to reproduce it manually — and hitting a real snag
Tried triggering it by hand: `ftp 10.10.10.20`, username `:)`, then
`nc 10.10.10.20 6200`. Got "Connection refused" — nothing was listening.
Retried the same sequence twice more, same result. Rebooted Metasploitable
entirely (via Proxmox console, since it has no home-network interface to
reach any other way) to rule out "the scanner already used up the
backdoor" — still failed identically after a clean reboot. Confirmed via
`nmap -p 21,6200` that port 21 was fine but 6200 was genuinely closed,
not filtered, nothing stuck.

Best explanation: OpenVAS's own "Active Check" for this finding likely
verifies it more conservatively (probably a version-banner match plus a
lighter check) rather than fully triggering the real, dangerous
root-shell backdoor itself — scanner vendors are often cautious about
that. Manually reproducing the raw protocol trigger by hand apparently
isn't quite enough, or isn't landing exactly right some other way that
wasn't fully diagnosed.

### Reaching for the right tool: Metasploit
Rather than keep guessing at the raw protocol by hand, used Metasploit
(`msfconsole`) and its dedicated, maintained module for this exact
vulnerability: `exploit/unix/ftp/vsftpd_234_backdoor`. First attempt
tried switching to a `cmd/unix/interact` payload — not a valid payload
name in this Metasploit version, corrected course and just filled in the
missing required field (`LHOST 10.10.10.10`) on the module's own default
payload (`cmd/linux/http/x86/meterpreter_reverse_tcp`) instead.

**It worked on the first real attempt with Metasploit** — backdoor
triggered, shell caught, payload fetched over HTTP, full Meterpreter
session established. `getuid` confirmed `root`; `sysinfo` confirmed the
target (`Ubuntu 8.04`, kernel `2.6.24-16-server`) matches Metasploitable2
exactly.

**Lesson:** a vulnerability scanner confirming something and a hand-rolled
manual reproduction attempt are not the same level of proof — and when
manual reproduction of a well-known exploit fails even after ruling out
obvious causes (state, timing, a stale target), reaching for the
purpose-built, tested tool (Metasploit's own module) is the right move
rather than continuing to debug a hand-rolled version of something
that's already been solved properly elsewhere.

### Checking whether Security Onion caught it
Went to Security Onion's Alerts page afterward. Three real alerts fired,
mapping cleanly onto distinct stages of the attack:
- **ET INFO Executable and linking format (ELF) file download** (high) —
  the compiled Meterpreter payload binary being transferred to the
  target — the delivery stage.
- **ET HUNTING curl User-Agent to Dotted Quad** (medium) — matches the
  payload's `FETCH_COMMAND: CURL` setting exactly; this signature watches
  for curl requests going to a raw IP rather than a hostname.
- **GPL ATTACK_RESPONSE id check returned root** (medium, fired 3 times)
  — a well known, old signature that watches for literal `id` command
  output showing `uid=0(root)` crossing the wire — post-exploitation
  confirmation, visible on the network, not just inside our own
  Meterpreter session.

No alert fired for the initial FTP trigger itself (the `:)` username) —
worth noting honestly as a real, normal limitation: not every step of an
attack has its own signature. Detection sometimes catches the
downstream effects of an attack rather than its opening move.

Didn't dig further into any single alert's raw fields this time — unlike
the earlier PostgreSQL alert triage, we already knew exactly what
happened (we ran the exploit ourselves) and the three alerts already
told a clear, internally consistent story without needing deeper
correlation work to confirm it. Recognizing when a story's already clear
enough not to need more digging is as much a real skill as knowing when
to dig deeper.

### Reflection
This was a genuinely complete loop: scanner finds a vulnerability →
understand what kind of vulnerability it actually is → attempt manual
verification → hit a real, honestly-diagnosed snag → reach for the
correct specialized tool instead of brute-forcing a workaround → confirm
successful exploitation directly → check whether the lab's own detection
stack caught it happening. Directly touches Vulnerability Management
(scan output, understanding a real CVE), Security Operations (using
SIEM/NSM tools to determine malicious activity), and Incident Response
(attack methodology, recognizing multi-stage incidents) all in one
session.
