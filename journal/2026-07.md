## 2026-07-16 — Security Onion install, part 1: planning and download

### Goal for this session
Start the Security Onion install on the Proxmox server (SV1), as the first real
piece of the home SOC lab.

### Deployment type decision
Compared Security Onion's deployment options against SV1's actual specs:

- SV1 specs: AMD Ryzen 7 3700X (16 cores), ~47GB RAM (~38GB free), multiple
  storage pools (`local` ~90GB free, `local-lvm` ~1.3TB free).
- Standalone deployment requirements: 4 CPU cores, 24GB RAM, 200GB disk.
- Hardware comfortably clears Standalone with room to spare, so went with
  Standalone instead of the lighter Import option.
- Standalone includes live network traffic monitoring/analysis, which is the

  the goal even though it's more setup work.

## 2026-07-17 — Security Onion install, part 2: network, VM, and installer

### Sniffing network bridge
Created `vmbr1` in Proxmox: Linux Bridge, no IPv4/CIDR (IP-less), no
physical NIC attached, autostart enabled. Fully isolated from the home
network by design — see the monitoring scope decision in
`docs/security-onion-install.md`.

### VM created
VM 101, name `security-onion`. 4 cores, 24GB RAM, 300GB disk on
`local-lvm` (bumped up from the 200GB Standalone minimum for headroom).
Two NICs: `net0` on `vmbr0` (management), `net1` on `vmbr1` (sniffing).
Firewall disabled on both. Full spec table and reasoning in
`docs/security-onion-install.md`.

### Installer, attempt 1 — cancelled
Went through the full `sosetup` wizard (Standalone, Standard internet,
hostname `securityonion`, `ens18` management via DHCP, `ens19` monitor,
Direct connection, email + password for the SOC web account, IP-based web

instead of the correct `192.168.8.0/24`. Cancelled before confirming

### Installer, attempt 2 — confirmed and running
Re-ran the wizard with the same choices, subnet corrected to
`192.168.8.0/24` this time. Confirmed at the summary screen. Install
kicked off (package sync, salt/minion setup, Elasticsearch, Kibana,
Kafka, Zeek, etc.) — this step takes a long time (30–60+ min) and is

### Management IP: DHCP + router reservation, not static-in-OS
Security Onion's installer recommends a static IP for management, but
this lab already manages Proxmox and `srv-infra` via DHCP + router-side
reservations rather than static IPs set inside the OS. Chose DHCP in the
installer and added a reservation in the GL-MT6000 router for the
management NIC's MAC (`bc:24:11:ef:96:7e`) → `192.168.8.12`, consistent
with the existing numbering (Proxmox `.10`, `srv-infra` `.11`, outside the
DHCP pool of `.100`–`.249`).

### QEMU Guest Agent
Per Security Onion's own Proxmox docs, installed the guest agent inside
the VM via SSH (while the main install was still running in the console):

```
sudo dnf install qemu-guest-agent
sudo systemctl enable qemu-guest-agent
sudo systemctl start qemu-guest-agent
```

`start` failed with a dependency error tied to a virtio-serial device
that Proxmox hadn't yet exposed to the VM. Enabled "Use QEMU Guest Agent"
in the VM's Options tab in Proxmox to fix this, but that setting doesn't
retroactively attach the device to an already-running VM — confirmed this
needs a full VM restart, not just an OS reboot. Deferred the restart until
the main Security Onion install finishes, to avoid interrupting it.

### Status at end of this entry
Security Onion install still running. SSH access to the VM confirmed
working (password auth) at `sysop@192.168.8.12`. Guest agent installed
and enabled but not yet running — pending VM restart.

### Install finished — with a scare
Came back after ~30 min away to a blank VNC console; reopening it showed
"Install had a problem. Please see /root/sosetup.log ... A summary of
errors can be found in /root/errors.log."

`errors.log` showed: `Error: Connection activation failed: Unknown error`

Switched to the SSH session (much easier to read/copy output than the
VNC console) and checked `sosetup.log` in full: the actual Salt state run
had already completed — **1667 states, 1667 succeeded, 0 failed** — before
this error occurred. The error was from a verification step after the
real work was done, most likely caused by a brief network blip during the
unattended window.

Confirmed everything was actually fine with `sudo so-status`: every core
service (Elasticsearch, Kibana, Zeek, Suricata, Strelka, Logstash, Redis,
etc.) reported running, several marked healthy, ending in Security
Onion's own "this onion is ready" success message.

**Takeaway:** don't panic-reinstall off a scary-looking final error message
— check the actual state summary and `so-status` first.

### First login
Logged into the web console at `https://192.168.8.12` with the admin
account created during setup. Self-signed cert warning, expected, clicked
through. Dashboard loaded. Core install is confirmed done and working.

### QEMU Guest Agent — resolved
Restarted the VM (clean shutdown, then start) so Proxmox could attach the
virtio-serial device the agent needs. Confirmed working:
`systemctl status qemu-guest-agent` shows `active (running)`, responding
to Proxmox's guest-ping calls, and Proxmox's UI now shows the VM's IP
directly.

### SSH key authentication — set up, with two real mistakes along the way
Generated an ED25519 keypair on the Windows machine:
```
ssh-keygen -t ed25519 -C "security-onion-lab"
```
Saved to the default location, set with a passphrase (decided a bare,
no-passphrase key wasn't worth the reduced security for convenience —
went with KeePassXC's SSH Agent feature instead, so the passphrase is
stored and auto-supplied without retyping it each time).

**KeePassXC SSH Agent setup:**
- Tools → Settings → SSH Agent → enabled, set to "Use OpenSSH" (not
  Pageant — Pageant is PuTTY's agent protocol; PowerShell's native `ssh`
  expects the Windows OpenSSH agent instead).
- Added the private key (`id_ed25519`, **not** the `.pub` file — the
  private key is what the agent needs to actually authenticate) to an
  entry's SSH Agent tab, with "Add key to agent when database is
  opened/unlocked" and "Remove key from agent when database is
  closed/locked" both checked.

**Mistake 1 — disabled the wrong service.** Earlier in this session,
disabled Windows' built-in `ssh-agent` service (OpenSSH Authentication
Agent) on the assumption it would conflict with KeePassXC. Backwards:
KeePassXC's "Use OpenSSH" mode doesn't replace that service, it feeds
keys *into* it. Without the service running, there's no agent pipe for
either KeePassXC or `ssh` to use at all. Fixed by re-enabling it from an
elevated PowerShell:
```
Set-Service -Name ssh-agent -StartupType Automatic
Start-Service ssh-agent
```
After starting the service, had to lock and re-unlock the KeePassXC
database to get it to push the key into the now-running agent — confirmed
with `ssh-add -l`, which showed the key's fingerprint once it worked.

**Mistake 2 — never actually copied the public key to the server.** Even
with the agent working locally (`ssh-add -l` showing the key), SSH kept
falling back to password auth. Turned out the earlier command to copy the
public key into the VM's `authorized_keys` had been given but never
confirmed run — got sidetracked into the KeePassXC setup before checking.
Verified directly on the server:
```
cat ~/.ssh/authorized_keys
# → No such file or directory
```
Ran the actual copy command (from the Windows side, not the SSH session):
```
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh sysop@192.168.8.12 "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```
One last password prompt to do the copy, then `ssh sysop@192.168.8.12`
logged straight in with no password. Confirmed working end to end.

**Lesson:** when chaining several setup steps together, verify each one
actually completed before moving to the next — two separate assumptions
(that a command had been run, and that a service should be disabled)
both turned out wrong and combined to make the actual root cause harder
to spot.

### Status at end of this session
Security Onion Standalone install complete and verified (`so-status`,
web console login). QEMU Guest Agent installed, enabled, and confirmed
running after a VM restart. SSH key authentication fully working via
KeePassXC's SSH Agent (OpenSSH mode), passphrase-protected key, no more
password prompts for `sysop@192.168.8.12`.

(Metasploitable2) plus an attacker VM (Kali Linux), both on `vmbr1`
alongside Security Onion's sniffing interface, so real scan/exploit

### Kali attacker VM — created, install in progress
Downloaded the official Kali 2026.2 standard installer ISO (not the net
installer — Kali's own docs note the net installer lacks drivers needed
for Proxmox's virtual network adapters):
```
https://cdimage.kali.org/kali-2026.2/kali-linux-2026.2-installer-amd64.iso
```
Checksum verified against official SHA256
(`6dbefacc95e3b556c19c48e8bae39b8b505e2d3a1aba0bfb7ab62b036c3d2ba3`) —
matched.

VM specs: `kali-attacker`, 2 cores, 4096 MiB RAM, 40GB disk on
`local-lvm`, single NIC on `vmbr1` (not `vmbr0` — this VM only needs to
reach its target and be visible to Security Onion's sniffing interface,
not the home network). QEMU Guest Agent checked at creation time this
round, to skip the restart dance that Security Onion's install required.

Network autoconfiguration failed during install (expected — `vmbr1` has
no DHCP server, by design, since it's an isolated bridge with no physical
NIC). Configured manually instead: assigned a new, separate private
range for this isolated lab segment — `10.10.10.0/24` — rather than
reusing the home network's `192.168.8.0/24`, to keep it clearly distinct.
Kali set to `10.10.10.10/24`, no gateway, no name server (isolated

Hostname `kali-attacker`, user `tyler` created. Disk partitioning: guided,
entire disk, all-in-one partition (40.7GB ext4 root + 2.2GB swap).

Install (package installation stage) in progress as of this entry.

### Kali install finished — static IP didn't persist
Install completed, GRUB installed to primary drive (only OS on the disk,
so straightforward). First login confirmed. `ip a` showed `eth0` up but
with **no IPv4 address** — the static `10.10.10.10/24` config entered
during the installer didn't carry over to the installed system.

Checked `/etc/network/interfaces` — only had loopback configured, nothing
for `eth0`. Turned out this system manages networking via NetworkManager
(`systemctl status NetworkManager` → active), not the classic
`interfaces` file, so the installer's static IP settings likely only
applied to the live installer session and never got written into
NetworkManager's config for the installed system.

Fixed directly with `nmcli` instead of fighting the installer:
```
sudo nmcli connection modify "Wired connection 1" ipv4.addresses 10.10.10.10/24 ipv4.method manual
sudo nmcli connection up "Wired connection 1"
```
Confirmed with `ip a` — `eth0` now shows `10.10.10.10/24` correctly, and
it persisted through a later reboot.

### Clipboard problem → added a management NIC
Couldn't copy/paste into the noVNC console (expected — noVNC doesn't
support host clipboard passthrough), which made typing commands
tedious and error-prone. Since `10.10.10.0/24` is fully isolated from the
home network, there was no way to SSH in from the Windows machine either
— no routing path exists between the two networks.

Fixed by mirroring Security Onion's own pattern: added a second NIC to
Kali on `vmbr0` (home network), purely for management/SSH access, keeping
`eth0`/`vmbr1` for the isolated lab work. Required a VM reboot for Kali to
see the new NIC. Came up as `eth1` with a DHCP address from the home
router: `192.168.8.249` (dynamic lease, not a reservation yet — consider
adding one later if this address needs to stay fixed, same as was done
for Security Onion's `.12`).

### SSH — enabled, then key auth set up (worked first try this time)
Unlike Security Onion, Kali doesn't run SSH by default (disabled out of
the box, consistent with being a security-testing distro rather than a
server OS). Enabled and started it:
```
sudo systemctl enable ssh
sudo systemctl start ssh
```
First connection attempt failed with "Connection refused" before this —
correctly diagnosed as "service not running" rather than a network
problem, since refused (vs. timeout) meant the path was fine but nothing
was listening on port 22.

Reused the same ED25519 keypair already generated for Security Onion
(no need for a second keypair — one identity, multiple servers) and
copied the public key over:
```
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh tyler@192.168.8.249 "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```
One password prompt to do the copy, then `ssh tyler@192.168.8.249` logged
straight in via the KeePassXC agent — no password, no repeated mistakes
this time (unlike the Security Onion round, where the same two steps got
done in the wrong order).

### Management IP reservation — done
Added a router reservation for `eth1`'s MAC (`bc:24:11:26:62:50`) →
`192.168.8.13`, continuing the existing numbering (Proxmox `.10`,
Security Onion `.12`, Kali `.13`). Forced the new lease by cycling the
connection from the SSH session itself:
```
sudo nmcli connection down "Wired connection 2" && sudo nmcli connection up "Wired connection 2"
```

This predictably dropped the SSH session mid-command (cycling the exact
interface the connection was using), but was harmless — reconnecting to
the new address (`ssh tyler@192.168.8.13`) worked immediately. Kali's
management address is now fixed, matching Security Onion's setup.

## 2026-07-18 — Metasploitable2, and a real multi-layered detection bug hunt

### SSH to the Proxmox host itself
Before starting on Metasploitable, set up key auth to SV1 (the Proxmox
host) too, not just the VMs — reused the same ED25519 keypair. `root@SV1`
accepted password auth already; copied the public key over the same way
as before and confirmed key login worked. Flagged as a genuinely
sensitive credential (root on the hypervisor underlies every other VM),
so extra emphasis on keeping the KeePassXC database itself well protected
going forward.

### Metasploitable2 download and import
Official source confirmed: SourceForge, `metasploitable-linux-2.0.0.zip`
(~800MB), default creds `msfadmin`/`msfadmin`. Per the project's own
warning, never expose this VM to an untrusted network — a non-issue here
since it lives on the isolated `vmbr1` segment by design, same as Kali.

Downloaded and extracted directly on the Proxmox host via SSH:
```
cd /var/lib/vz/template/iso
wget https://sourceforge.net/projects/metasploitable/files/Metasploitable2/metasploitable-linux-2.0.0.zip/download -O metasploitable-linux-2.0.0.zip
apt install -y unzip   # not installed by default on Proxmox
unzip metasploitable-linux-2.0.0.zip
```
Extracted into `Metasploitable2-Linux/`, containing `Metasploitable.vmdk`
(the actual disk image) plus some VMware-specific files that aren't
needed for Proxmox.

Created VM 104 (`metasploitable`) with no disk (deleted the auto-created
one), 1 core, 512MB RAM, network on `vmbr1`, QEMU agent checked at
creation. Imported the VMDK as a new disk:
```
qm importdisk 104 Metasploitable.vmdk local-lvm
```
Attached the resulting disk via the Hardware tab (auto-assigned as
`scsi0`), boot order auto-updated to boot from it.

### Bug 1 — wrong disk controller for a 2012-era kernel
First boot failed: `ALERT! /dev/mapper/metasploitable-root does not
exist. Dropping to a shell!`, and `cat /proc/partitions` inside the
rescue shell showed **no disks detected at all**. Root cause: Proxmox's
default SCSI controller for this VM was VirtIO SCSI, which didn't exist
as a common option when this VM image was built (~2012) — its old
kernel/initramfs has no driver for it, so it couldn't see the disk to
even attempt mounting it.

Fixed by moving the disk from `scsi0` to an IDE bus instead (universally
supported, even by very old kernels), via the Proxmox shell rather than
the GUI (no in-place bus-conversion option in the UI, only detach/
reattach):
```
qm set 104 --ide0 local-lvm:vm-104-disk-0
qm set 104 --delete scsi0
qm set 104 --boot order=ide0
```
Booted cleanly after this — full ASCII banner, services starting
(Tomcat, Apache), login prompt reached.

### Networking: same static-IP pattern, older tooling
Logged in (`msfadmin`/`msfadmin`), `eth0` had no IPv4 (only link-local
IPv6) despite being up — this OS predates NetworkManager/netplan
entirely, so it just needed the classic Debian
`/etc/network/interfaces` file edited directly:
```
auto eth0
iface eth0 inet static
    address 10.10.10.20
    netmask 255.255.255.0
```
`sudo ifdown eth0 && sudo ifup eth0` applied it. Confirmed with
`ifconfig eth0` (this old system doesn't have the modern `ip` command)
— `inet addr:10.10.10.20` correct, and RX/TX counters showed it was
actively passing traffic.

### Connectivity confirmed — but nothing showed up in Security Onion
`ping -c 4 10.10.10.20` from Kali: 0% packet loss, working perfectly.
But searching Security Onion's Hunt page for `10.10.10.20` after running
an nmap scan (`nmap -sV 10.10.10.20`) returned **zero results**. This
kicked off a real, multi-layered troubleshooting chain — worth writing
up in full since each layer looked like "the" answer before turning out
to be one piece of a bigger picture.

**Layer 1 — search technique, ruled out.** A bare IP as a Hunt query is
valid Onion Query Language (matches across all fields), confirmed this
was working syntax; it just wasn't matching anything real. A plain
`10.10.10.20` search briefly returned results that turned out to be
Security Onion's *own* web dashboard API logs (`event.dataset:
soc.server`) — the browser's own traffic to the SOC interface, not
network capture data. Switched to a more specific query
(`source.ip:10.10.10.10 OR destination.ip:10.10.10.20`) to avoid that
noise going forward.

**Layer 2 — the sniffing interface was never actually active.**
Checked `ip a` on Security Onion directly: `ens19` (the monitor NIC) was
`UP` at the link level but missing the `PROMISC` flag, and `bond0` (the
bonded interface Security Onion actually captures from — this is normal
Security Onion design, not a misconfiguration) showed `NO-CARRIER` /
`state DOWN`. `nmcli con show` revealed why: a connection profile named
`bond0-slave-ens19` existed (the intended binding) but was never
activated — `DEVICE` column showed `--`.

Tried activating it manually (`sudo nmcli con up bond0-slave-ens19`) —
failed with `Error: Connection activation failed: Unknown error`. This
is the **exact same error text** seen during the original install (and
dismissed at the time as a false alarm after `so-status` came back
clean) — in hindsight, that dismissal may have been premature; this may
be the same underlying issue surfacing again rather than a one-off.

Checked `journalctl -u NetworkManager` for the real reason this time
(the connection-specific filter returned nothing useful, but a broader
`--since "5 minutes ago"` filter did): `mtu greater than device maximum
... setting MTU to requested size is not possible`. Security Onion's own
config (`/opt/so/saltstack/local/pillar/minions/securityonion_
standalone.sls`) specifies `mtu: 9000` for the monitor interface (jumbo
frames — matches Security Onion's own documented Proxmox guidance), but
`vmbr1` and Security Onion's `net1` NIC were still at Proxmox's default
1500, so the bond-slave attachment kept failing at the MTU-setting step.

Fixed by setting `vmbr1`'s MTU to 9000 in Proxmox (Network → vmbr1 →
Edit), and leaving Security Onion's `net1` NIC set to "Same as bridge"
(inherits 9000 automatically, avoids the two settings drifting out of
sync later). **This required a full VM restart** to take effect — a
live Proxmox-side MTU change doesn't reach an already-running guest, same
lesson as the QEMU guest agent issue from the initial install. After
restart: `ens19` showed `mtu 9000`, and `nmcli con up bond0-slave-ens19`
succeeded — `ens19` now shows `PROMISC` and `master bond0`, `bond0` shows
`UP, LOWER_UP`.

**Layer 3 — sniffing was active, but still saw almost nothing.** With
the bond genuinely up, re-ran the ping/nmap test. `tcpdump -i bond0 -n`
on Security Onion captured exactly one packet (an ARP broadcast) and
some unrelated DHCP broadcast noise from Metasploitable — but *zero* of
the actual ICMP ping traffic between Kali and Metasploitable, even
though the ping itself succeeded perfectly (0% loss) on Kali's end.

This is the real structural gap, and it's about how Linux bridges work,
not a Security Onion problem: `vmbr1` is a standard Linux bridge, which
forwards unicast traffic directly port-to-port based on MAC learning —
it does **not** flood unicast traffic to every port the way a hub would.
Being in promiscuous mode only helps Security Onion's port see whatever
traffic *actually arrives* there; a two-way conversation between Kali's
and Metasploitable's ports simply never gets forwarded to a third,
uninvolved port unless something explicitly mirrors it there. This is
the fundamental reason a plain bridge isn't sufficient for a "sniffing"
setup — real SPAN/port-mirror configuration is a deliberate, additional
step, not a side effect of promiscuous mode.

Fixed with explicit Linux `tc` (traffic control) mirroring rules on the
Proxmox host, mirroring both VMs' tap interfaces to Security Onion's tap
interface. Identified the actual host-side tap device names first
(Proxmox's `tap<VMID>i<index>` convention):
```
ip a | grep -B 1 "master vmbr1"
# → tap103i0 (Kali), tap104i0 (Metasploitable), tap101i1 (Security Onion)
```
Then added mirroring rules on both VM ports, targeting Security Onion's
port:
```
tc qdisc add dev tap103i0 handle ffff: ingress
tc filter add dev tap103i0 parent ffff: protocol all u32 match u32 0 0 action mirred egress mirror dev tap101i1

tc qdisc add dev tap104i0 handle ffff: ingress
tc filter add dev tap104i0 parent ffff: protocol all u32 match u32 0 0 action mirred egress mirror dev tap101i1
```
Retested immediately: `tcpdump -i bond0 -n host 10.10.10.20` on Security
Onion now showed the full ICMP conversation (all 4 echo request/reply
pairs) while pinging from Kali. Confirmed working.

**Note for later:** these `tc` rules are applied directly via the Proxmox
shell and are **not persistent** — they won't survive a host reboot.
Should be scripted/automated (e.g. a boot-time hook or Proxmox hookscript)
if this lab is going to be rebuilt or the host restarted; for now,
documenting the exact commands here is the fallback if they need to be
reapplied manually.

### End-to-end confirmed
Re-ran `nmap -sV 10.10.10.20` from Kali and searched Security Onion's
Hunt page (`source.ip:10.10.10.10 OR destination.ip:10.10.10.20`).
Real results this time: dozens of `zeek.conn` and `zeek.weird` entries
across every port nmap probed, and a genuine Suricata alert —
**"ET SCAN Suspicious inbound to PostgreSQL port 5432"** (category:
Potentially Bad Traffic) — triggered by the scan hitting Metasploitable's
PostgreSQL port. The full detection pipeline (isolated network → traffic
generation → mirroring → Zeek/Suricata analysis → SOC dashboard) is
confirmed working end to end.

Home SOC lab is functionally complete: Security Onion, Kali attacker VM,
and Metasploitable2 target VM all running on the isolated `vmbr1` segment,
traffic properly mirrored to Security Onion's sniffing interface, real
