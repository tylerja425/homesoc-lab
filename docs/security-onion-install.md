# Security Onion Install — Standalone on Proxmox

This document describes how Security Onion 2.4.211 was installed as a
Standalone deployment on the home lab's Proxmox server (SV1). It's written
as a reference for reproducing the setup or understanding the decisions
behind it, not as a step by step tutorial for a general audience.

## Why Standalone

Security Onion offers several deployment types (Import, Eval, Standalone,
Distributed, Desktop). Standalone was chosen because:

- It supports live network traffic monitoring and analysis, which is the
  actual skill this lab is meant to build: writing detections against real
  traffic, not just reviewing pre-captured logs.
- The host hardware comfortably exceeds Standalone's minimums (4 cores,
  24GB RAM, 200GB disk):
  - CPU: AMD Ryzen 7 3700X, 16 cores
  - RAM: ~47GB total, ~38GB free at time of install
  - Disk: 1.3TB+ free on the `local-lvm` storage pool

## Monitoring scope: isolated virtual lab

Security Onion's sniffing interface needs to see mirrored network traffic.
Two options were considered:

1. **Monitor the whole home network** — requires a managed switch with
   port mirroring/SPAN support, or a physical network tap. More "realistic"
   traffic, but touches real household devices and their traffic, and adds
   hardware/config complexity.
2. **Monitor an isolated virtual lab network** — a Proxmox-only bridge with
   no physical NIC attached, traffic generated by other VMs on that same
   bridge. Safer, simpler, matches how most guided SOC lab material (e.g.
   TryHackMe's SOC Level 1 path) assumes the environment is built.

Went with option 2. Can add a home-network tap as a phase 2 later if
wanted; the underlying analyst skills (SIEM use, detection writing, alert
triage) transfer identically either way.

## Proxmox network setup

Created a second, IP-less Linux Bridge dedicated to sniffing:

- Name: `vmbr1`
- IPv4/CIDR: none (bridge doesn't need its own address — it's for
  mirrored/internal traffic only, not routing)
- No physical NIC attached (keeps it fully isolated from the home network)
- Autostart enabled

The VM's existing management traffic continues to use the host's normal
bridge, `vmbr0`.

| Memory         | 24576 MiB (24GB)                        |
| Disk           | 300GB, on `local-lvm` (thin provisioned)|
| NIC 1 (net0)   | `vmbr0` — management, firewall off      |
| NIC 2 (net1)   | `vmbr1` — sniffing/monitor, firewall off|

Firewall was disabled on both NICs: Proxmox's per-NIC firewall isn't
needed here (Security Onion is the thing doing security monitoring) and
can interfere with normal operation, especially on the sniffing interface.

Promiscuous mode on the sniffing NIC (required for it to see traffic not
addressed to it) is handled automatically by Security Onion's own setup
wizard when the monitor interface is selected — no manual Proxmox-side
config needed for this.

## Installer walkthrough

The Security Onion ISO boots into a kickstart-driven base OS install
(Rocky Linux under the hood via the Anaconda installer), then reboots into
Security Onion's own interactive setup wizard (`sosetup`).

Base OS install: username `sysop` created as the administrative Linux
account, password set and stored in a password manager (not written down
anywhere in this repo).

Security Onion setup wizard choices:

| Prompt                          | Choice                          | Reasoning |
|----------------------------------|----------------------------------|-----------|
| Installation type               | STANDALONE                       | See above |
| Internet access                 | Standard                         | Node has direct internet access, no proxy in this network |
| Hostname                        | `securityonion`                  | Default, fine for a single-node home lab |
| Management NIC                  | `ens18`                           | Matches Proxmox `net0` (`vmbr0`) by MAC address |
| Management IP method            | DHCP                             | See note below — matches existing home lab convention |
| Docker IP range                 | Default                          | No conflicts with existing networks |
| Monitor interface               | `ens19`                           | Matches Proxmox `net1` (`vmbr1`) by MAC address |
| Internet connection method      | Direct                           | No proxy server in this environment |
| Web interface admin account     | Personal email + password        | Login for the SOC web console (separate from the `sysop` Linux account) |
| Web interface access method     | IP                                | No internal DNS set up to resolve a hostname reliably |
| Allow web access                | Yes                               | Required to actually use the dashboard |
| Allowed IP/subnet for web access| `192.168.8.0/24`                 | Allows any device on the home LAN to reach the dashboard; blocks anything outside the network |
| SOC Telemetry                   | Enabled                          | Anonymous UI usage stats only — no security data or logs sent. Supports the open source project. Adjustable later in SOC Configuration. |

### Note on DHCP vs static management IP

Security Onion's installer recommends a static IP for the management
interface. This lab already manages other hosts (Proxmox itself, the
`srv-infra` VM) via DHCP + router-side reservations rather than IPs
configured inside the OS. To stay consistent with that convention:

- Chose **DHCP** in the Security Onion installer.
- Added a DHCP reservation in the home router (GL-MT6000) for the VM's
  management NIC MAC address (`bc:24:11:ef:96:7e`), pinning it to
  `192.168.8.12` — outside the router's dynamic pool (`.100`–`.249`) and
  consistent with the existing numbering (Proxmox `.10`, `srv-infra` `.11`).

This achieves the same practical outcome as a static IP (address never
changes, no collision risk) while matching how the rest of the home lab
is already managed.

### Mistake caught during setup: wrong subnet on first attempt

On the first pass through the wizard, the "allowed IP or subnet" field was
entered as `192.168.9.0/24` (typo) instead of the correct `192.168.8.0/24`
— the actual home network's subnet. Caught at the final confirmation
screen before committing. Cancelled the wizard (`sosetup` reruns
automatically on next login) and re-ran it correctly. No lasting effect;
this happened before the install actually began.

## QEMU Guest Agent

Security Onion's own Proxmox documentation recommends installing the
QEMU Guest Agent post-install, so this was done as part of initial setup
rather than treated as optional:

```
sudo dnf install qemu-guest-agent
sudo systemctl enable qemu-guest-agent
sudo systemctl start qemu-guest-agent
```

Also required on the Proxmox side: VM → Options → QEMU Guest Agent → "Use
QEMU Guest Agent" checked. Note that enabling this option does not
retroactively add the virtio-serial device to an already-running VM — the
guest agent service will fail to start (`dependency job failed`) until the
VM is fully restarted (not just the guest OS rebooted) so Proxmox can
attach the new virtual hardware.

_Status: package installed and enabled; service start pending a full VM
restart after the main Security Onion install completes._

## Install completion and a false-alarm error

The Salt-driven install itself completed cleanly: 1667 states run, 1667
succeeded, 0 failed. A final verification step then threw:

```
Error: Connection activation failed: Unknown error
```

This looked alarming (Security Onion's own setup wizard reports "Install
had a problem" and points to `/root/sosetup.log` / `/root/errors.log`),
but the actual install log showed the failure happened *after* all 1667
states had already succeeded — it was a verification/network-check step,
not part of the real configuration work. Likely cause: a brief network
interruption during a ~30 minute window the host was unattended.

Confirmed the install was actually fine by running:

```
sudo so-status
```

All core services (Elasticsearch, Kibana, Zeek, Suricata, Strelka,
Logstash, Redis, etc.) reported `running`, several explicitly `healthy`.
Security Onion's own status check gave its standard all-clear message.

**Lesson for next time:** an error at the very end of `sosetup` doesn't
necessarily mean the install failed — check `so-status` and the tail of
`sosetup.log` before assuming a reinstall is needed.
